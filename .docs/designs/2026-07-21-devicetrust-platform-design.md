## Problem Statement

Reaching a self-hosted OpenClaw agent from a phone fails at the plumbing, not the chat.
The official OpenClaw mobile apps (shipped 2026-06-29) launched to "can't pair at all"
reviews and 2.2 stars on Android. Live testing on this project hit the identical wall:
the cryptography worked first try, but ephemeral tunnel URLs, minutes-lived setup codes,
and a manual approval dance took three rounds and a human relay to complete. The product
gap is trust + reachability that never fails — not another chat UI.

## Demand Evidence

- Verified: the founder — built and uses the app daily against their own gateway;
  today's pairing ordeal is first-hand evidence of the pain.
- Strong proxy: the official app's launch reviews — users "unable to pair the app with
  their Gateway at all" (Android Authority, 2026-06-29 coverage), 2.2★ Android.
- Unverified (flagged honestly as inferred): broader community demand. No named external
  user yet — this is the assignment.

## Status Quo

Self-hosters reach their agent today via: (1) the official mobile app — free, MIT, but
launch-broken at pairing; (2) Slack/Telegram channel plugins — works, but routes a
private agent through third-party servers and buries it in work chat; (3) webchat over
a manually-managed tunnel — requires tunnel-fu and dies on every cloudflared restart.
Cost of the status quo: private traffic through relays they don't control, or periodic
manual re-setup (new tunnel URL → new setup code → re-approval) that non-engineers
cannot perform.

## Target User & Narrowest Wedge

Target user (archetype, verified n=1): an engineer running an autonomous OpenClaw agent
on a home server, away from keyboard half the day, who loses steering/visibility while
mobile. Consequence: agent runs unattended or stalls on approvals until they're back at
a laptop. Assignment (below) is to name 3 more.

Narrowest wedge: **pairing that works the first time.** That alone is the exact step
the official app fails at, and it is fully client-scoped except for the upstream
dependencies named below. The managed named tunnel (stable wss:// URL across restarts)
is deliberately NOT part of the wedge — it is Approach B's first milestone.

Mechanics of "one-tap": the gateway's `openclaw qr` already emits a QR whose payload is
base64url JSON `{url, bootstrapToken}` (verified live). The client scans it, derives
host + bootstrap token, and runs the signed connect + PAIRING_REQUIRED retry loop
already implemented in Settings. What the client CANNOT fix alone: setup-code lifetime
(minutes) and the operator-approval step — see Upstream Dependencies.

## Constraints

- Solo founder, long-game posture (explicitly rejected urgency premise P3).
- iOS-first; zero third-party deps in the client (repo rule).
- Gateway protocol-v4 is upstream-controlled (MIT); protocol changes can land anytime.
- Trust-sensitive market: self-hosters chose self-hosting to AVOID relays. Honest trust
  model: Ed25519 device keys secure AUTH end-to-end (keys never leave the device), but
  protocol-v4 has no end-to-end payload encryption — whoever terminates TLS on the
  tunnel can see chat traffic. Therefore the managed tier runs on the CUSTOMER'S
  Cloudflare account (vendor orchestrates config via API token, traffic never touches
  vendor infrastructure). To be explicit: Cloudflare still terminates TLS and can see
  traffic on ANY tunnel — the customer-account model removes the vendor, not
  Cloudflare, from the trust boundary. Removing Cloudflare too requires protocol-v4
  payload encryption (upstream) and is out of scope. A vendor-proxied tier would trade
  further trust for convenience and is out of scope until a customer explicitly asks.

## Upstream Dependencies (gateway-side, not ours to control)

First-time-right pairing requires gateway cooperation on two points the client cannot
fix: (1) setup-code lifetime — minutes-lived codes expire before a normal person
finishes; (2) the manual operator-approval step. Strategy (REVISED, eng review
2026-07-21 D6): NO upstream PRs — the wedge is client-side and contributing TTL/
auto-approve fixes would repair the official app's pairing too. Mitigations are ours
alone: QR scan compresses entry to seconds (defeats the TTL in practice), the
approval-wait UX shows the operator the exact approve command, and the companion CLI
on the gateway box (`devicetrust pair`) is the escape hatch if gateway friction
proves fatal. Approach C is dead, not folded.

## Premises

1. AGREED — A free official MIT app means competing head-on as "another OpenClaw chat
   client" is dead; differentiation must be reliability/connectivity or a neglected
   segment.
2. AGREED — The real product is the trust + reachability layer (DeviceTrust + managed
   tunnel); the chat UI is the demo of it.
3. REVISED — Long-game infrastructure play: the official app's stumbles are evidence of
   the need, not a closing window. (Founder explicitly rejected the urgency framing.)

## Approaches Considered

Effort anchors (solo founder): S ≤ 2 weeks, M ≤ 6 weeks, XL ≥ 3 months.

### Approach A: The Reliable Client (minimal viable)
Premium third-party client; DeviceTrust internal; sell "pairing that works first try."
Effort S, Risk Med. Pros: ships now, fastest to real users. Cons: commoditizing lane;
official-app improvements erode the edge.

### Approach B: DeviceTrust Platform (ideal architecture)
The connectivity+trust product: one-tap pairing, managed named tunnels, device fleet
dashboard, revocation, push relay. App is the reference client. Monetize the managed
service (the Path-D control plane the repo's SyncSource seam anticipates).
Effort XL, Risk Med-High. Pros: solves the reported failure; survives official-app
fixes; recurring-revenue shape. Cons: long build; must win self-hosters' trust.

### Approach C: Own the Standard (lateral)
Contribute pairing UX + DeviceTrust hardening upstream (MIT), earn distribution, launch
the managed service later. Effort M. Pros: cheapest validation. Cons: no moat until
phase 2; upstream may slow-walk.

## Recommended Approach

**A→B sequence (approved by founder; resequenced by eng review 2026-07-21 D5).**
Ship the CURRENT build to TestFlight first — tester #1 gets the already-working
pairing+chat before any refactor lands (outside-voice finding 8, founder-accepted).
The GatewayConnection actor and PairingService extraction land AFTER tester-1
feedback; the QR scanner lands BEFORE TestFlight (it IS the wedge deliverable).
B's first milestone (REVISED, D7): `devicetrust setup` — a LOCAL CLI on the gateway
box that configures a named tunnel on the customer's own Cloudflare account. No
hosted control plane, no API-token custody (holding the token would reinstate the
vendor in the trust boundary). Monetization is discovered from real usage, not
assumed. Fleet/pairing dashboard and push relay stay BEHIND the demand gate. Note:
push notifications violate the repo's v1 scope rule by design; that rule is
overridden only at B kickoff, not before. Naming decision locked this session: the auth subsystem is **DeviceTrust**
(descriptive, whole-subsystem). Component mapping in the repo: `DeviceIdentity` +
`DeviceAuth` (DeviceAuth.swift — keys, deviceId, v3 signing), pairing flow
(SettingsView.pair + GatewayWSSyncSource.connectOnce), token lifecycle
(SettingsStore.deviceToken via KeychainService). Names slot in unchanged.

## Success Criteria

- A (wedge = pairing): 3 named external users complete pairing on the first attempt
  without founder assistance; TestFlight retention past week 1. Why credible without
  upstream fixes: QR scanning compresses code entry to seconds (defeating the
  minutes-TTL in practice), and a self-hoster IS their own operator — the
  approval-wait UX showing the exact approve command closes the loop solo.
- B milestone 1a (reachability): a gateway reachable at ONE stable wss:// URL for 30
  days across cloudflared and gateway restarts.
- B milestone 1b (credential durability): device tokens survive gateway restarts with
  zero re-pairing (device keys already do — verified by design).
- Demand gate before any B feature beyond the tunnel CLI: at least one person states
  what they would pay for (REVISED D7 — the $5/mo hosted hypothesis is retired; free
  local CLI first, willingness-to-pay discovered from usage).

## Design Review Addendum (2026-07-21)

Approved visual spec: `.docs/designs/assets/2026-07-21-pairing-states.png`
(6 frames: demo-banner bridge, entry, scanning, waiting-for-approval, paired,
failed-expired; exact Theme.swift tokens). T1 implements these screens.

### Decisions (all resolved)
1. **Settings hierarchy is pairing-state-dependent (1A).** Unpaired: pairing hero
   (Scan Setup Code) on top, host/token/model collapsed under an "Advanced"
   disclosure (the QR carries the host — manual fields are the fallback, not the
   front door). Paired: compact status row first, fields below.
2. **First-run bridge = demo-mode banner (2A).** Persistent amber banner above the
   chat input in demo mode: "Demo mode — pair your gateway to talk to your real
   agent →"; tap opens pairing; disappears when paired. Demo path itself unchanged
   (protected by CLAUDE.md).
3. **A11y spec (3A), folded into T1:** state transitions announced via
   AccessibilityNotification; step-ladder rows carry VoiceOver labels; 44pt min
   touch targets; mono text respects Dynamic Type; paste path documented as the
   non-camera route.

### Interaction states (what the user SEES)
| State | Badge | Ladder | Primary content | Primary action |
|---|---|---|---|---|
| Demo chat | amber banner | — | "Demo mode — pair your gateway…" | tap → pairing |
| Entry | — | — | why-pairing card + `openclaw qr` hint | Scan Setup Code (paste + Advanced below) |
| Scanning | — | — | camera frame + auto-fill explainer | (scan); "Enter code manually" escape |
| Connecting | green spinner | step 1 active | — | Cancel |
| Waiting approval | amber WAITING ON GATEWAY | 1✓ 2● 3○ | approve command in copyable code block + retry countdown | Cancel |
| Paired | green PAIRED | 1✓ 2✓ 3✓ | key-in-Keychain + revocation reassurance | **Start Chatting →** |
| Failed: expired | red CODE EXPIRED | fail dot | cause + `openclaw qr` fix + "under 10 seconds" | Scan New Code |
| Failed: camera denied | — | — | permission explainer | Paste code instead (+ Settings deep link) |
| Failed: timeout | red | 2 fail | "approval didn't arrive" + approve command | Retry |

Copy rules: utility language, mono; every failure names its cause AND its fix;
recovery is always the primary button — never a dead-end error.
